# Plan: Let dev (debug) builds coexist with Release builds on-device

## Context

When a Release APK built by GitHub CI is already installed, a local development
install fails:

```
INSTALL_FAILED_VERSION_DOWNGRADE: Update version code 1 is older than current 2
```

**Why it happens:** Both the CI Release APK and the local debug APK share the
same `applicationId` (`com.autom8ed.rgbsunglassesapp`), so Android treats them as
the *same app*. The Release APK carries a high `versionCode` (CI computes
`MAJOR*10000 + MINOR*100 + PATCH`, e.g. `app-v1.0.2` → `10002`), while the local
debug build is hard-coded to `versionCode 1`. Installing the lower code over the
higher one is rejected as a downgrade.

**Goal:** Make the development build install and run *alongside* the Release
build as a separate, visually distinguishable app — without disturbing the CI
release pipeline.

**Key constraint:** `android/` is **gitignored** (`app/.gitignore`) and is
regenerated by `npx expo prebuild --clean` both locally and in CI
(`.github/workflows/app-release.yml:80`). Therefore **any hand-edit to
`android/app/build.gradle` or to `android/.../res/` would be wiped**. The fix
must be expressed through Expo's config system so it is re-applied on every
prebuild.

## Approach: a local Expo config plugin that adds a `.dev` debug variant

Give the **debug** build type a distinct `applicationId` via
`applicationIdSuffix ".dev"`, plus a "(Dev)" label and a tinted adaptive-icon
background. Because the dev app has a different package name, the versionCode
downgrade check no longer applies (it is per-package), and both apps live on the
device at once.

- Dev/debug app id → `com.autom8ed.rgbsunglassesapp.dev`, label "RGB Sunglasses (Dev)"
- Release app id → `com.autom8ed.rgbsunglassesapp` (unchanged)

The CI workflow builds `assembleRelease` only, so the debug-scoped changes are
inert for releases — no workflow edits required.

## Files

### New: `app/plugins/withDevVariant.js` (local config plugin, CommonJS)

A single plugin exporting the default, composed of three sub-mods from
`@expo/config-plugins` (available transitively via `expo`):

1. **`withAppBuildGradle`** — inject into the `buildTypes { debug { … } }` block
   only (target the `buildTypes`-scoped `debug {`, *not* the identically-named
   `signingConfigs { debug {`), adding:
   ```gradle
   applicationIdSuffix ".dev"
   ```
   Use a tightly-anchored regex such as `/buildTypes\s*\{\s*debug\s*\{/` and
   insert immediately after it. Make the insertion idempotent (skip if
   `applicationIdSuffix ".dev"` already present).

2. **`withDangerousMod` (android)** — write a **debug source set** so the dev
   build overrides label/icon via Android resource merging (variant > main).
   The mod runs after Expo has generated the main resources and writes into the
   regenerated tree each prebuild:
   - `android/app/src/debug/res/values/strings.xml` → `<string name="app_name">RGB Sunglasses (Dev)</string>`
     (overrides the `app_name` Expo writes to `src/main/res/values/strings.xml`).
   - **Tinted icon:** read the generated
     `android/app/src/main/res/mipmap-anydpi-v26/ic_launcher.xml` (and
     `ic_launcher_round.xml`), copy them into `src/debug/res/mipmap-anydpi-v26/`,
     and rewrite only the `<background>` reference to point at a solid dev color
     (e.g. `@color/iconBackgroundDev`); leave the `<foreground>`/`<monochrome>`
     refs pointing at the shared `@mipmap/...` assets. Write
     `src/debug/res/values/colors.xml` defining `iconBackgroundDev` (an amber
     such as `#FFB300`). Reading-and-rewriting the generated XML (rather than
     hardcoding it) keeps the plugin resilient to Expo's icon output changing.
     This produces a clearly different-looking dev icon (solid tinted background
     behind the same foreground) without regenerating per-density raster assets.

   Guard every write so the mod is safe to re-run, and only act when the
   expected main resources exist.

### Modified: `app/app.json`

Add the local plugin to the `plugins` array (path-string form), e.g.:
```json
"./plugins/withDevVariant"
```
No other app.json changes; `version`/`versionCode` stay as-is (CI still sets
them at release time).

### Unchanged (verify only)
- `.github/workflows/app-release.yml` — builds `assembleRelease`; unaffected.
- `android/app/build.gradle` — not edited directly; the plugin re-applies the
  suffix on each prebuild.

## Verification

1. **Regenerate native project:** from `app/`, run
   `npx expo prebuild --platform android --clean`.
2. **Confirm gradle injection:** `android/app/build.gradle` debug build type now
   contains `applicationIdSuffix ".dev"` (and the release block is untouched).
3. **Confirm debug source set:** `android/app/src/debug/res/values/strings.xml`,
   `colors.xml`, and `mipmap-anydpi-v26/ic_launcher*.xml` exist with the dev
   label and tinted background.
4. **Coexistence test (end-to-end):**
   - Install a Release APK (CI artifact, or `./gradlew assembleRelease`) — appears as "RGB Sunglasses".
   - Then `npx expo run:android` (debug) — installs as a *second* app
     "RGB Sunglasses (Dev)" with the tinted icon; **no** `INSTALL_FAILED_VERSION_DOWNGRADE`.
   - `adb shell pm list packages | grep rgbsunglasses` shows both
     `com.autom8ed.rgbsunglassesapp` and `com.autom8ed.rgbsunglassesapp.dev`.
5. **Release sanity:** `./gradlew assembleRelease` still yields applicationId
   `com.autom8ed.rgbsunglassesapp`, label "RGB Sunglasses", original icon.

## Notes / trade-offs

- **Shared URL scheme:** both apps register scheme `rgbsunglassesapp`; with both
  installed, Android may prompt which app to open a deep link. Acceptable for dev.
- **First-install of dev on a clean device** is unaffected; this only adds a
  parallel package.
- If the read-and-rewrite of the adaptive-icon XML proves brittle on this Expo
  version, fall back to overriding just `src/debug/res/values/colors.xml`
  `iconBackground` (the color Expo already references) — a smaller but still
  visible tint change.
